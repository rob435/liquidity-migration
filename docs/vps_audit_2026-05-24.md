## 4. Order error handling — `submitted_unconfirmed` pattern

In `event_demo` order placement, a `place_order` exception is treated as "submitted_unconfirmed" (not "failed"), with the rationale documented inline:

> "The exception may be a lost response AFTER Bybit accepted the order, so the entry is ledgered submitted_unconfirmed (a pending status): next cycle `_reconcile_pending_order_fills` probes get_trade_history and adopts a real fill, or the pending guard expires it. A 'failed' status is skipped by reconciliation and would orphan a live position."

This is correct behavior for the lost-response failure mode. Same pattern applied to fill-confirmation timeouts.

## 5. Dual-sleeve `ws_risk` routing (recent fix `fe61446`)

Single risk engine reads both ledgers (`event_demo_trades` + `long_native_demo_trades`), tagged by a `sleeve` column. The recent fix added sleeve tagging on all write paths.

**Read path** — `_read_trades_combined` / `_read_orders_combined`:
- Each side tagged with `_ensure_sleeve_column` before concat
- Long-root read wrapped in try/except — failures are "fail open" (treated as empty), correct: the risk engine should not crash because one ledger is locked for write

**Write path** — `_write_trade_rows_routed` / `_write_order_rows_routed`:
- Each row inspected: sleeve=='long' AND long_root present → long ledger; else → short ledger
- Defensive `assert self.long_root is not None` in long-branch
- Default for unknown sleeve: `"short"` (historical default)

**Sleeve tagging fallback chain** — `_tag_sleeve_from_trades`:
1. Use row's existing sleeve if non-empty
2. Else look up trade_id in combined ledger
3. Else look up symbol in combined ledger
4. Else fall back to `"short"`

This 4-level chain is the load-bearing defense against silent misrouting. The fix correctly applies it at every dual-sleeve write site (the fix commit message lists "orphan adoption + exits + repairs" as the three previously-untagged paths).

**One observation, not a bug:** the final fallback to `"short"` could mis-route an unknown-symbol long-sleeve row if both indices miss it. The prior fix made sure all upstream writes tag the sleeve before reaching this fallback, so it's protective for unexpected cases, not the primary path. No action needed unless a future signal pattern can emit untagged long-sleeve rows.

## 7. Paper-vs-demo reconciliation tool

[reconciliation.py](liquidity_migration/reconciliation.py) (280 lines, full read):
- Pairs trades by (symbol, side, entry_ts ± tolerance) using **global gap-first** assignment (smallest entry-time gaps win first, not greedy per-demo nearest-time). This is correct — a greedy pass would let an earlier demo trade consume a paper trade that's a tighter match for a later one and bias slippage.
- Computes entry slippage in bps (side-aware: lower fill is adverse for shorts; higher fill is adverse for longs)
- Computes exit slippage in bps (closing-side-aware)
- Per-trade and summary outputs
- Markdown report renderer included

The tool itself is sound. Running it requires the live paper/demo ledgers (`event_demo_trades` table) under both data roots, which live on the VPS. Once pulled locally or run on the VPS:

```bash
python -m liquidity_migration reconcile-paper-demo \
  --paper-root /opt/liquidity-migration/data/bybit-paper-event \
  --demo-root /opt/liquidity-migration/data/bybit-demo-event
```
